## 位置づけとゴール

台帳の Phase 1 分割の後半。ゴールは1つ: **「書いた画」と「模範の画」の近さを数字で出し、どの画がどうちがうかを特定できると確認すること**。

これは一発で決まらない種類の作業である。だから成果物は「完成した判定」ではなく、**実機でしきい値を調整するための道具一式**。合否・しきい値の確定は実機検証後に Chat 側で行う。

## 設計判断（Chat 決定済み。行き詰まったら勝手に方式転換せず停止して相談 — §7-4）

1. **前処理**: KanjiVG の path と、ユーザーの筆跡を、どちらも同じ座標系に正規化してから比べる
   - 模範: 各画の path を等間隔リサンプリングした固定点数の座標列に変換(点数は 16〜32 の範囲で Code が決めてよい。決めたら記録)
   - ユーザー: 生ストロークを同じ点数にリサンプリング
   - 正規化はマス座標基準（KanjiVG は 109×109 viewBox。マスとのスケール合わせを最初に固定する）
2. **比較は画単位・筆順どおり**: i 番目に書いた画は i 番目の模範画とだけ比べる（照合哲学=筆順厳格）
   - 画数が模範と違う時点で、その旨を返す（多い/少ない）
3. **画ごとのスコア**は最低限これで開始:
   - 対応点間の平均距離（形・位置のずれ）
   - 始点→終点の方向の一致（書く向きの検出。右と左の1画目問題、逆向きに書いた画を捕まえる）
   - 2要素の重み付き合成で 0〜100。重みとしきい値は定数を1箇所にまとめ、調整画面から動かせるように
4. **字全体の判定**: 全画のスコアとしきい値から、○ / おしい（対象画を特定）/ ×
5. **とめ・はね・はらいの厳密判定はやらない**（判定哲学=字形寛容）。画の存在・順序・向き・大まかな形だけを見る

## タスク1: エンジン実装

- `js/matcher.js`（ビューアや後続フェーズから使い回せる独立モジュール。DOM 非依存）
- 入力: ユーザー筆跡（ストローク配列）＋ 模範（kanji-g1.json の1字）
- 出力: `{ verdict, perStroke: [{score, dist, dirOk}], ... }` の構造化結果
- 模範のリサンプリング結果はビルド時生成でも実行時変換でもよいが、方式と理由を完了報告に書く（1026字×容量への影響を含めて）

## タスク2: 調整ページ（このフェーズの本体）

- `match-test.html`: Phase 0 の書き味マスを土台に、
  - 80字から出題字を選択（またはランダム）
  - 書くと1画ごとに即時スコア表示（画リストに数値と○△×、対象画のハイライト）
  - しきい値・重みのスライダー（変更が即反映）
  - 現在の設定値を画面に表示（検証結果の報告に使うため）
- バージョン表示（★反映確認の教訓）
- 装飾不要。これは調整用の道具であり子ども向け UI は Phase 2

## タスク3: 自己テスト（実機検証の前に機械で潰す）

最低限これを自動化して通すこと:

- 模範データ自身をユーザー入力として与える → 全画が高スコアで○になる（自明ケース）
- 画の順序を入れ替えた入力（右の1画目と2画目を交換など）→ 該当画が低スコアになる
- 1画欠け・1画過剰 → 画数エラーが返る

## 検証（神田さんの作業。手順を提示して停止）

1. 実機で match-test.html を開く（バージョン確認）
2. ていねいに正しく書く → ○が出るか（出ないなら、しきい値が厳しすぎ）
3. わざと筆順を替える（右・左が好例）→ 該当画が指摘されるか
4. 子どもを想定して速く雑に書く → ○のままか（×になるなら字形に厳しすぎ）
5. 体感としきい値の当たり所を Chat に報告 → しきい値の確定と合否は Chat 側

## 停止ポイント

- タスク1〜3 完了で停止し、完了報告（採った方式・設定値・自己テスト結果）
- 実機検証の結果待ち。台帳更新は検証後に Chat 承認を得てから

## やらないこと

筆跡保存 / 判定結果の保存 / 子ども向け文言（「おしい！」等の演出は Phase 2）/ とめはね・筆圧 / 何の字か当てる認識 / 複数字の連続出題 / PWA
