対象: Claude Code / 推奨モデル: Fable 5（日本語の読み・送り仮名の処理。ただし小規模）
前提: 台帳・決定事項リストを読むこと。既存 v0.8。

## 見つかった不具合（背景）

よみ問題の3択で、選択肢の送り仮名がバラバラ（例:「生まれる」等の送り仮名付きと「山」「川」等の漢字だけ＝読みのみが混在）。
これは見た目の問題であると同時に**公正性バグ**: 正解の字は文脈に合わせ送り仮名が付くのに、誤答肢は他の字から機械生成で送り仮名が付かないため、**送り仮名の有無で正解が透けて見える**。連濁バグと同型（機械生成が意味・語形を見ていない）。

## 方針（Chat決定）: 選択肢は「読みだけ」に揃える＝方向A

よみ問題で問うのは「その字をなんと読むか」であり、送り仮名はかき問題の領分。
全選択肢を送り仮名なしの読みに統一する（「生まれる」→「うまれる」の"読み"部分の扱いは下記で確認）。

- 誤答肢に自然な送り仮名を付ける方向Bは採らない（誤答ごとの送り仮名生成が新たな不自然さ＝連濁型の穴を生むため）

## タスク0: 現状確認（実装前に必ず・停止して報告）

よみ問題の画面構成を確認し、報告すること。方針の詰めがこれに依存する:

- **(あ)** 問題文に対象語が文脈として表示されているか（例「いもうとが生まれた。」の文が出て、赤下線/空欄の読みを3択で選ぶ形か）
- **(い)** それとも字だけを見せて読みを問う形か

判定基準:

- (あ)＝文脈ありの場合: 選択肢は純粋な読みでよい。送り仮名の活用語尾も含めた「よみ」を、送り仮名の"送り"部分を除いた形で揃える。具体的には、選択肢に出す文字列は KANJIDIC2 のドット前（語幹）＋文脈上の活用がある場合はその読み全体のうち、どの選択肢も同じ規則で生成されていること。要は「ある選択肢だけ送り仮名付き・別は無し」という不揃いを無くすのが主眼
- (い)＝字のみの場合: 送り仮名なしの読みだけでは意味が取りづらい懸念がある。この場合は Chat に相談（画面に文脈を足す設計変更が要るかもしれない）

## タスク1: 選択肢生成の統一（タスク0の報告後・(あ)の場合）

- `js/exam.js`（または該当の選択肢生成箇所）で、正解肢・誤答肢を同一の規則で整形する
- 送り仮名の扱いを1つの関数に集約（例: readingForChoice(reading) が KANJIDIC2 形式の読みから選択肢表示文字列を一貫生成）。正解も誤答もこの関数を通す
- 不揃い（片方だけ送り仮名付き）が構造的に起きないようにする
